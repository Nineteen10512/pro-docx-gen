Step 1: pro_docx_gen/self_audit.py — NEW MODULE (630 lines)
────────────────────────────────────────────────────────────────────
Public API:
  audit_template_file(path)            -> AuditReport
  audit_template(module)               -> AuditReport
  audit_all_templates(templates_dir=None) -> List[AuditReport]
  audit_docx_output(docx_path)         -> AuditReport
  AuditReport / Violation dataclasses
  main(argv=None) CLI with --templates / --docx / --templates-dir

Mechanism:
  - ast-based scanning of template files (not regex; structurally impossible
    to false-positive on docstrings/comments/constant tables).
  - Walks every register(DOCXTemplate(...)) call's kwargs and Dict literals
    recursively. Tracks field names that are heading/title/h1..h6/text/body/
    muted. Emits Violation if the literal matches FORBIDDEN_HEADING_HEXES.
  - DOCX output audit uses xml.etree.ElementTree to walk word/document.xml,
    extract pStyle of each paragraph, gather run colours via <w:color> tags,
    and checks: (a) no heading uses a forbidden hex, (b) all paragraphs at
    the same level share one colour.
  - "防复发": any future template that injects #D4AF37 / #E0E7FF / #64FFDA
    will trip the gate. Any future renderer regression that lets a heading
    run override the style colour to a forbidden hex will trip the gate.

Verification (synthetic negative tests):
  Test 1: Template with heading=#D4AF37 + text=#E0E7FF
          -> 2 violations, status=fail ✓
  Test 2: Template with heading=#1A1A1A
          -> 0 violations, status=pass ✓
  Test 3: DOCX with one H1 in #D4AF37 and one H1 in #1A1A1A
          -> 2 violations: forbidden hex + colour inconsistency ✓
  Test 4: DOCX with all H1 in #1A1A1A (allowed) and one H2 in #333333
          -> 0 violations, status=pass ✓

Step 2: pro_docx_gen/engine/chart_renderer.py — MODIFIED (560 lines)

      chart_spec, theme_tokens, output_path, dpi=None,

Changes:
  - Added "page-fit contract" auto-shrink: when img_h > available_h * 0.9,
    the figure is scaled proportionally (width and height both shrink) to
    fit the remaining page area. chart_spec gets the new metadata keys
    _figure_height_in / _figure_width_in / _figure_scale_applied /
    _compact_mode so the renderer can pick the right paragraph spacing.
  - compact_mode flag is stashed on chart_spec and consumed by
    pro_docx_gen.engine.renderer to set space_before/space_after=0 and
    keep_with_next=True on the caption paragraph. Default behaviour
    (compact_mode=False) is unchanged.

Verification:
  No available_height: width=6.27in, height=4.702in, scale=1.0
  available_height=3.0in:
    width=3.6in, height=2.7in, scale=0.5742  (aspect ratio preserved)
    compact_mode=True propagated correctly
  100% backward compatible (new params have defaults; existing callers
  in renderer.py updated; no other call sites exist).

Step 3: pro_docx_gen/figure_preflight.py — MODIFIED (319 lines)
────────────────────────────────────────────────────────────────────
New public API:
  AVAILABLE_HEIGHT_FILL_RATIO = 0.90         (constant)
  compute_available_height_inches(tokens, *, consumed_inches=0.0) -> float
  _emu_to_inches(value) -> float             (private helper)

Mechanism:
  - Reads tokens["spacing"]["page_height"], tokens["spacing"]["page_margin"],
    tokens["page"]["margin_top"]/"margin_bottom"/"gutter".
  - Heuristic EMU → inches conversion: anything > 30 inches is treated as
    EMU (so layout.content_width which is already inches is not double-
    converted). Safe defaults: A4 page height (11.69 in), 1.0 in top/bottom
    margin.
  - Returns max(page_h - top - bottom - gutter - consumed, 0.5) so we
    never promise a sub-half-inch insertion.

Verification (synthetic):
  tokens with A4 page + 1in margin -> available = 9.69in
  with consumed=3.0in -> available = 6.69in
  Both numbers pass through chart_renderer correctly.

Changes:

    missing_document_xml errors (defensive: don't double-report).

    lazily (so the gate stays usable from a clean checkout), runs
    audit_all_templates(), prints the verdict to stderr, and exits 1
    if any template fails the v1.6.6 colour audit. This is the
    "防复发" mechanism: forbidden templates cannot ship.
