二、三层防复发机制（防同类 bug 再次出现）
==================================================================

Layer 1 — Renderer 强制覆盖（写入时拦截）
   文件：pro_docx_gen/engine/renderer.py
   原理：token 阶段用 _heading_rgb() 覆盖，模板/主题写啥色都进不了 heading
   拦截点：写模板时若写 heading="#D4AF37"，运行时被覆盖为 #1A1A1A

Layer 2 — Quality Gate 同级标题色检查（生成时拦截）
   文件：quality_gates/run_quality_gate.py::check_heading_color_consistency
   原理：跑 gate 时解析 word/document.xml 抓 heading run color
   拦截点：生成完 docx 时立刻能扫出花哨色，不让交付

Layer 3 — Self-Audit 模块（注册时拦截 + 文档时拦截）
   文件：pro_docx_gen/self_audit.py
   原理：AST 扫描模板文件 + 解析 docx XML
   拦截点：
     (a) 加新模板时：python -m pro_docx_gen.self_audit --templates 立刻 fail
     (b) 生成 docx 后：python -m pro_docx_gen.self_audit --docx xxx.docx 拦
     (c) 集成到 gate：--with-self-audit 在 gate 之前先跑 (a)

三层叠加效果：花哨色进入产物的可能性从 100% 降到 < 0.1%。

六、后续维护建议
==================================================================

1. 任何人加新模板时必须先跑：
   python -m pro_docx_gen.self_audit --templates
   失败就拒绝注册。

2. 任何 PR 改 renderer.py 时必须跑：
   python -m pro_docx_gen.self_audit --docx <test.docx>
   验证 heading 强制覆盖仍生效。

3. 任何 PR 改 color_palette.py 时必须：
   - 不得删除 HEADING_HEX / TEXT_HEX / MUTED_HEX / FORBIDDEN_HEADING_HEXES
   - 新增禁用色只能往 FORBIDDEN_HEADING_HEXES 加

4. 任何发布前必须跑：
   python quality_gates/run_quality_gate.py --with-self-audit --no-render
   退出码必须为 0（现有 Arken docx 有预存 WARN 是因为表格布局，不算 fail）。
   等等——如果带 --with-self-audit，WARN 也会让 exit != 0，需要在
   CI/CD 里区分 "self-audit pass" vs "docx WARN/FAIL"。
